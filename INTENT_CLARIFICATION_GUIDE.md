# 意图澄清和安全检测功能指南

## 概述

为了提高文档编辑的准确性和安全性，系统新增了三大安全机制：

1. **意图澄清机制** - 当用户请求模糊时，先询问确认
2. **语义冲突检测** - 检测修改前后内容的主题是否一致
3. **修改范围限制** - 对于大范围替换，要求用户明确确认

## 功能详解

### 1. 意图澄清机制

#### 触发场景

**跨段落引用检测**

当用户消息包含以下模式时，系统会请求澄清：
- "改成第X条"
- "参考第X章"
- "像第X段那样"
- "按照第X节"

**示例**：
```
用户: "把第五条知识产权部分改一下，改成第四条说的对"

系统响应:
{
  "status": "need_clarification",
  "message": "检测到您引用了其他段落",
  "clarification": {
    "type": "cross_reference",
    "question": "您是想：\n1. 完全替换为被引用段落的内容？\n2. 参考被引用段落的格式/结构来改写？\n3. 参考被引用段落的表述方式？",
    "options": [
      {"id": "replace_content", "label": "完全替换内容"},
      {"id": "reference_format", "label": "参考格式结构"},
      {"id": "reference_style", "label": "参考表述方式"}
    ],
    "severity": "high"
  }
}
```

**模糊表达检测**

使用 LLM 检测以下模糊请求：
- 使用"改一下"、"优化一下"等笼统表达
- 没有明确说明要改成什么样
- 使用"更好"、"更合适"等主观词汇但没有具体标准

**示例**：
```
用户: "把这段改得更好一些"

系统响应:
{
  "status": "need_clarification",
  "message": "您的请求可能不够明确",
  "clarification": {
    "type": "ambiguous",
    "reason": "使用了主观词汇'更好'但没有具体标准",
    "question": "为了更好地帮您修改，请明确：\n- 希望改成什么风格？（正式/通俗/简洁）\n- 需要保留哪些关键信息？\n- 有什么具体的改进方向？",
    "severity": "medium"
  }
}
```

**大范围修改检测**

对以下操作要求确认：
- 批量替换操作（`multi_replace`）
- 删除操作（`delete`）

**示例**：
```
用户: "删除所有关于预算的内容"

系统响应:
{
  "status": "need_clarification",
  "message": "这是一个删除操作",
  "clarification": {
    "type": "delete_operation",
    "question": "删除后无法恢复（除非回滚版本），确认删除？",
    "severity": "high"
  }
}
```

### 2. 语义冲突检测

#### 检测内容

系统会检测以下类型的语义冲突：

1. **主题完全不同**
   - 例如：技术规范 → 付款条款
   - 严重程度：高

2. **逻辑矛盾**
   - 例如：必须 → 禁止
   - 严重程度：高

3. **关键信息丢失**
   - 例如：删除了重要的约束条件
   - 严重程度：中

4. **上下文不连贯**
   - 例如：修改后与前后段落脱节
   - 严重程度：中

#### 工作流程

```
1. 用户提交修改请求
   ↓
2. 系统生成编辑计划
   ↓
3. 生成预览时检测语义冲突
   ↓
4. 如果检测到高严重度冲突
   ↓
5. 在预览中显示警告
   ↓
6. 用户确认后才能应用
```

#### 示例

**场景：主题不符**

```
原内容: "第四条 验收标准\n\n4.1 甲方应在收到乙方书面验收通知后15个工作日内组织验收"

新内容: "第四条 知识产权\n\n4.1 本项目开发成果的知识产权归甲方所有"

系统检测:
{
  "type": "semantic_conflict",
  "conflict_type": "主题不符",
  "severity": "high",
  "message": "修改前后内容主题完全不同（验收标准 → 知识产权）",
  "suggestion": "建议保持原有主题，或明确说明要更改主题"
}

预览消息:
"请确认以下修改

⚠️ 修改前后内容主题完全不同（验收标准 → 知识产权）"
```

**场景：格式调整（不冲突）**

```
原内容: "第五条 知识产权\n\n本项目开发成果的知识产权归甲方所有"

新内容: "第五条 知识产权\n\n5.1 本项目开发成果的知识产权归甲方所有\n5.2 乙方保证交付成果不侵犯任何第三方的知识产权"

系统检测:
{
  "has_conflict": false
}

预览消息:
"请确认以下修改"
```

### 3. 跨段落引用解析

#### 功能说明

当用户引用其他段落时，系统会：
1. 自动识别被引用的段落编号
2. 查找对应的段落内容
3. 将引用内容作为上下文传给 LLM
4. 指导 LLM 正确理解用户意图

#### 支持的引用格式

- 中文数字：第一条、第二章、第三节
- 阿拉伯数字：第1条、第2章、第3节
- 混合：第十一条、第12章

#### 工作原理

```python
用户: "把第五条改成第四条那样"
      ↓
系统识别: 引用了"第四条"
      ↓
查找第四条内容
      ↓
传给 LLM 的提示:

"""
目标块内容: [第五条的内容]

参考内容（第4条）: [第四条的内容]

注意：用户说"改成第4条"，通常是指：
1. 参考其格式和结构（如编号方式、条款组织）
2. 参考其表述风格（如用词、语气）
3. 而不是完全替换内容

请保持目标块的主题，只借鉴参考内容的形式。
"""
```

## 使用示例

### 示例 1：跨段落引用

**用户请求**：
```
"把第五条知识产权部分改一下，改成第四条说的对"
```

**系统流程**：

1. **意图澄清**
```json
{
  "status": "need_clarification",
  "message": "检测到您引用了其他段落",
  "clarification": {
    "question": "您是想：\n1. 完全替换为被引用段落的内容？\n2. 参考被引用段落的格式/结构来改写？\n3. 参考被引用段落的表述方式？",
    "options": [...]
  }
}
```

2. **用户选择**：选择"参考格式结构"

3. **系统执行**：
   - 查找第四条内容
   - 分析第四条的格式（如 4.1, 4.2, 4.3 的编号方式）
   - 将第五条改写为类似格式（5.1, 5.2, 5.3）
   - 保持知识产权的主题内容

4. **预览确认**：
```
第五条 知识产权

修改前:
本项目开发成果的知识产权归甲方所有

修改后:
5.1 本项目开发成果的知识产权归甲方所有
5.2 乙方保证交付成果不侵犯任何第三方的知识产权
5.3 乙方有权将用于本项目的技术用于其他项目
```

### 示例 2：语义冲突检测

**用户请求**：
```
"把第三条项目周期改成付款条款"
```

**系统流程**：

1. **定位成功**：找到第三条

2. **生成编辑计划**：将第三条内容替换

3. **语义冲突检测**：
```json
{
  "type": "semantic_conflict",
  "conflict_type": "主题不符",
  "severity": "high",
  "message": "修改前后内容主题完全不同（项目周期 → 付款条款）"
}
```

4. **预览显示警告**：
```
请确认以下修改

⚠️ 修改前后内容主题完全不同（项目周期 → 付款条款）

第三条 项目周期

修改前:
自合同签订之日起12个月...

修改后:
甲方通期付款0.05%的违约金...
```

5. **用户确认**：用户看到警告后可以：
   - 取消修改
   - 确认继续（如果确实想改主题）

### 示例 3：模糊请求澄清

**用户请求**：

**系统响应**：
```json
{

  }
```

**用户重新表述**：

## 配置选项

### 启用/禁用功能

在 `.env` 文件中配置：

```bash
# 启用意图澄清
ENABLE_INTENT_CLARIFICATION=true

# 启用语义冲突检测
ENABLE_SEMANTIC_CONFLICT_DETECTION=true

# 冲突检测严格程度（low/medium/high）
CONFLICT_DETECTION_THRESHOLD=medium
```

### 调整检测灵敏度

在 `app/nodes/intent_clarifier.py` 中调整：

```python
# 跨段落引用检测的正则表达式
patterns = [
    r'改成第[一二三四五六七八九十\d]+条',
    r'参考第[一二三四五六七八九十\d]+[条章节段]',
    # 添加更多模式...
]

# LLM 检测模糊度的温度参数
response = self.llm.chat_completion_json(messages, temperature=0.3)
```

## 最佳实践

### 用户侧

1. **明确表达意图**
   - ✅ "把第五条改成 5.1, 5.2, 5.3 的格式"
   - ❌ "把第五条改一下"

2. **说明引用方式**
   - ✅ "参考第四条的编号格式来改第五条"
   - ❌ "改成第四条那样"

3. **提供具体标准**
   - ✅ "改得更简洁，控制在200字以内"
   - ❌ "改得更好一些"

### 开发侧

1. **合理设置阈值**
   - 不要过于敏感，避免频繁打断用户
   - 也不要过于宽松，确保安全性

2. **提供清晰的选项**
   - 澄清问题时给出具体的选项
   - 避免让用户再次用自然语言描述

3. **记录用户选择**
   - 学习用户的偏好
   - 优化后续的意图理解

## 故障排查

### 问题：系统总是要求澄清

**原因**：检测过于敏感

**解决**：
1. 调整正则表达式，减少误判
2. 提高 LLM 检测的温度参数
3. 增加白名单，跳过常见的明确表达

### 问题：语义冲突检测不准确

**原因**：LLM 理解偏差

**解决**：
1. 优化 system prompt
2. 提供更多上下文
3. 调整严重程度阈值

### 问题：引用解析失败

**原因**：段落编号格式不匹配

**解决**：
1. 检查文档的实际编号格式
2. 添加更多正则表达式模式
3. 使用 LLM 辅助解析复杂格式

## 未来改进

2. **多轮对话**

## 总结

这三大安全机制显著提升了系统的可靠性：

1. **意图澄清** - 减少误解，提高准确率
2. **语义冲突检测** - 防止错误修改，保护文档完整性
3. **跨段落引用** - 正确理解复杂需求，提升用户体验

通过这些机制，系统能够更好地理解用户意图，避免错误操作，提供更安全、更智能的文档编辑体验。
